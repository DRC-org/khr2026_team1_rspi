# Gazeboシミューレーション 座標系不一致とAMCLトラブルの原因と現状

## 現在直面している現象
1. **初期位置は合っているが、動き出すと点群とマップがずれる**
   - 起動直後の Rviz では、ロボットは 2D マップ上の正しい位置（X=0.8, Y=1.0）付近に表示される。
   - レーザースキャン（点群）も壁と重なっているように見える。
   - しかし、ナビゲーションを開始し、ロボットが実際に移動（前進・回転など）を始めると、瞬く間にロボットの推定位置と点群がマップの境界線（壁）から逸脱してしまう。
2. **挙動が左右逆転している（または操作と進行方向が不一致）**
   - Rviz 上で指定した方向や経路（青い線）に対して、ロボットが逆方向へ向かおうとしたり、不自然な旋回を続ける。

## 根本原因の考察（なぜずれるのか）

現在のシミュレーション環境には、「Gazebo上の絶対座標系（World）」と「ROS2 / Nav2 上のマップ座標系（Map）」、「オドメトリ座標系（Odom）」が混在しており、**X軸方向の向き、あるいは反転の解釈が各層でバラバラになっている** ことが全ての元凶です。

### 1. `MecanumDrive` プラグインと URDF の不整合
- 車輪の配置（X, Yの正負）と、`MecanumDrive` プラグインが出力する `odom` トピックの進行方向が一致していない可能性があります。
- Gazebo内で前進命令（`cmd_vel` の X が正）を出したとき、物理的にロボットが +X 方向に進んでいるのか、-X 方向に進んでいるのかが混乱しています。

### 2. `/tf` ツリーの競合とオドメトリの逆転
- `ros_gz_bridge` 経由で Gazebo が直接 `/tf`（`odom -> base_link`）を配信しています。
- これに対し、AMCL はレーザー（`/scan`）とマップ（`field.pgm`）の照合から `map -> odom` の補正を計算します。
- **「Gazeboが流す移動方向（物理）」と「AMCLがレーザーから計算する移動方向（視覚）」が正反対（あるいは90度ズレ）**になっているため、動いた瞬間に AMCL が「想定と逆に動いた！」とパニックを起こし、補正をかけすぎてマップから逸脱します。

### 3. 初期位置変更（X=2.7 と X=0.8 の往復）による後遺症
- 本来のマップ（`field.pgm`）は `origin: [0,0,0]` で設定されています。
- 「左右逆な気がする」という違和感から X 座標を 3.5 から引いた値（2.7）へ反転させましたが、これにより「2次元マップ画像上の壁」と「Gazebo上の3Dの壁」の相対位置関係が破綻しました。
- 現在は全てをオリジナル（0.8 系）に戻しましたが、そもそも「初期状態での LiDAR の向き（前後左右）」と「マップの向き」が合っているかの検証が不十分なまま動かしてしまっています。

## 解決に向けた次のステップ（ロードマップ）

この問題はパッチワーク（部分的な座標修正）では解決できません。以下の順序で、一番下の層（物理・TF）から一つずつデバッグして積み上げる必要があります。

### Step 1: 物理レイヤーとオドメトリの確認（Nav2 を切って検証）
- **目的**: ロボット本体が本当に意図した方向に前進・回転するか確認する。
- **方法**: 
  - Nav2（AMCLなど）を立ち上げず、Gazebo と Rviz だけを起動する。
  - `teleop_twist_keyboard` などで手動操作し、「前進指令（+X）」でロボットが画面上の +X 方向へ進み、レーザースキャンも正しく追従するか（壁の距離が正しく縮むか）を確認する。
  - もしこの時点で LiDAR の壁が逆方向に動くなら、URDF のタイヤ配置や LiDAR の向き（`roll, pitch, yaw`）が間違っています。

### Step 2: TF ブリッジ設定の クリーンアップ
- **目的**: `/tf` の出処を一本化する。
- **方法**:
  - Gazeboのブリッジから `/tf` の転送を完全に切る。
  - 代わりに ROS 2 側の `robot_localization` ノード（EKF）を使い、Gazebo の `odom` や `imu` から ROS 専用のきれいな `odom -> base_link` の TF を生成させるのが ROS2 のスタンダードな設計です。

### Step 3: マップと AMCL の同期
- Step 1, 2 で「手動で動かせば Rviz 上でロボットと LiDAR の点群が滑らかに連動する」状態を作った上で、初めて AMCL を起動します。
- この状態で初期位置（X=0.8, Y=1.0）を与えれば、移動しても点群がマップの壁から剥がれることはなくなります。
